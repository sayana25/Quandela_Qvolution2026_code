🔴 Redundant reservoir forward passes Critical
The reservoir and projection matrix are both frozen, yet model(X_train) re-runs the full quantum circuit every epoch. Pre-compute features once before the training loop.
Estimated speedup: 100–500× on training loop
🔴 No reproducible projection seed Critical
The random projection torch.randn(...) is not seeded separately, so results differ across runs despite torch.manual_seed(0) being set earlier — any intervening torch ops shift the RNG state.
Impact: reproducibility
🟡 Flat learning rate for 500 epochs Medium
A constant LR of 0.01 for the full run often overshoots near convergence. Adding a CosineAnnealingLR or ReduceLROnPlateau scheduler typically improves final accuracy by 1–3%.
🟡 No early stopping Medium
Training continues all 500 epochs even after the loss plateaus. A simple patience-based early stop on validation loss prevents overfitting and wastes no compute.
Can cut epochs by 50–80%
🟢 Unverified complex output assumption Low
The code assumes reservoir(x) returns complex tensors and splits real/imag. MerLin's AMPLITUDES strategy may return real-only values, making imag all zeros and halving your effective feature space for free.
🟢 Logging inside training loop Low
The if epoch % 50 == 0 block recomputes logits.argmax(1) which is already available. Minor, but once pre-computing is done, the loop structure should be simplified.

Key Optimization: Pre-compute Frozen Features

The most impactful change. Since both the reservoir and the projection are non-trainable, run them once before the training loop and store the result. The linear layer trains on those cached features.
✕ Original — runs reservoir every epoch
# ❌ No pre-computation
# forward() runs full quantum
# circuit on every epoch

class MerlinReservoirClassifier(nn.Module):
    def forward(self, x):
        q = self.reservoir(x)  # expensive!
        q_feat = torch.cat(
            [torch.real(q),
             torch.imag(q)], dim=1
        ).float()
        z_like = q_feat @ self.proj  # also fixed!
        return self.linear(z_like)

✓ Optimized — pre-compute once
# ✅ Extract features ONCE before loop

def extract_features(reservoir, proj, X):
    with torch.no_grad():
        q = reservoir(X)
        # assert if real-only to save dim
        if torch.is_complex(q):
            feat = torch.cat(
                [q.real, q.imag], dim=1
            ).float()
        else:
            feat = q.float()
        return feat @ proj

X_train_feat = extract_features(
    reservoir, proj, X_train)
X_test_feat  = extract_features(
    reservoir, proj, X_test)

# Simplified model — just a linear layer
class FastReadout(nn.Module):
    def forward(self, x):
        return self.linear(x)  # instant!

# Training loop — no reservoir call
for epoch in range(epochs):
    optimizer.zero_grad()
    logits = linear(X_train_feat)  # ← fast
    loss = criterion(logits, y_train)
    loss.backward()
    optimizer.step()

Optimization 2: LR Scheduler + Early Stopping

✕ Original
optimizer = torch.optim.Adam(
    model.linear.parameters(), lr=0.01
)

for epoch in range(500):
    # no scheduler, no stopping
    ...
    loss.backward()
    optimizer.step()
    # trains all 500 epochs always
✓ Optimized
optimizer = torch.optim.Adam(
    [linear.weight, linear.bias], lr=0.01
)
scheduler = torch.optim.lr_scheduler
    .CosineAnnealingLR(optimizer, T_max=500)
best_loss, patience, wait = inf, 30, 0

for epoch in range(500):
    ...
    loss.backward()
    optimizer.step()
    scheduler.step()
    if loss < best_loss - 1e-4:
        best_loss, wait = loss, 0
    else:
        wait += 1
    if wait >= patience: break

Why the reservoir bottleneck is so severe

Quandela's MerLin photonic simulation involves computing the permanent of large matrices — this is #P-hard complexity. Even on 4 modes with 3 photons, each forward pass involves combinatorial computations. Running this 500× per epoch on 114 training samples is 57,000 permanent computations that produce the exact same result every time.

After pre-computing: the training loop is just pure matrix multiplications on 4-dimensional features — essentially free.

Feature dimension check

With AMPLITUDES measurement strategy, MerLin returns the complex probability amplitudes of the Fock basis states. For n_modes=4, n_photons=3, the number of basis states is C(4+3-1, 3) = C(6,3) = 20, so output_size = 20.

If MerLin returns complex tensors: feat_dim = 40. If already real: feat_dim = 20. Add assert q.is_complex(), "MerLin returned real — halve feat_dim" to verify.
Recommended order — start with #1, it's free performance.
1. Pre-compute frozen features ✓ highest priority
Add an extract_features() call for both X_train and X_test before the training loop. Replace model(X_train) inside the loop with linear(X_train_feat). This eliminates all quantum circuit calls during training.
2. Fix projection reproducibility
Use torch.Generator().manual_seed(42) when initializing proj. This ensures the projection matrix is identical across runs, isolating reservoir quality from initialization noise.
3. Add cosine LR scheduler
Wrap Adam with CosineAnnealingLR(optimizer, T_max=epochs) and call scheduler.step() after each optimizer step. Gentle decay in the last 30% of training helps the linear readout settle better.
4. Add patience-based early stopping
Track best training loss with a patience counter (suggest patience=30–50 for Iris). Break the loop when loss hasn't improved by 1e-4 for patience consecutive epochs.
5. Verify complex vs real output
Add print("Complex output:", torch.is_complex(reservoir(X_train[:1]))) after model init. If False, change feat_dim = reservoir.output_size and skip the real/imag split — you get a denser projection.
6. (Optional) Tune reservoir hyperparameters
Once baseline works, experiment with reservoir_layers = 3 or 4, and n_photons = 4. More photons → larger Fock space → richer features, but slower pre-compute. Since pre-compute is now one-time cost, this tradeoff is more acceptable.
